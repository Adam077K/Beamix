## Fixes Applied

### FIX-01 — Middleware Restored (CRITICAL)
**File:** `saas-platform/src/middleware.ts` (new file)
**Bug:** `src/middleware.ts` was deleted during Stripe → Paddle migration, leaving no Supabase session refresh mechanism. Without middleware, JWT tokens were not refreshed on each request, causing sessions to silently expire mid-use even though the `(protected)/layout.tsx` redirect guard still functioned.
**Fix:** Restored middleware using `@supabase/ssr` `createServerClient` with the canonical cookie-forwarding pattern. Calls `supabase.auth.getUser()` to refresh the session token on every non-static request. No redirect logic — that stays in the layout. Includes an env-var guard to prevent crashes during build without `.env.local`.

### FIX-02 — BusinessProfileTab Loads Real Data (HIGH)
**Files:**
- `saas-platform/src/app/api/businesses/primary/route.ts` — added `GET` handler
- `saas-platform/src/app/(protected)/dashboard/settings/page.tsx` — fixed `BusinessProfileTab`

**Bug:** `BusinessProfileTab` initialised all form fields with hardcoded placeholder values (`'My Business'`, `'https://example.com'`, `'tech'`, `'Tel Aviv, Israel'`). Saving the form would overwrite the user's real business data with these defaults if they hadn't manually cleared each field.

**Fix:**
1. Added `GET /api/businesses/primary` endpoint that returns the authenticated user's primary business fields: `name, website_url, industry, location, description, services`.
2. Added `useEffect` in `BusinessProfileTab` that fetches from the endpoint on mount, populates all form state from the response, and shows a loading skeleton while the request is in flight. Falls back gracefully (form stays editable) if the fetch fails.

### FIX-03 — Stripe Columns Migration (MEDIUM)
**File:** `saas-platform/supabase/migrations/20260302_remove_stripe_columns.sql` (new file)
**Bug:** After migrating from Stripe to Paddle, Stripe-specific columns (`stripe_customer_id`, `stripe_subscription_id`, `stripe_price_id`, `stripe_current_period_end`, `stripe_cancel_at_period_end`) remained in the `subscriptions` table schema. The migration hadn't been applied, leaving dead columns and schema drift.
**Fix:** Created a migration file with `IF EXISTS` guards that safely drops all Stripe columns from `subscriptions`, `user_profiles`, and `businesses` tables. Safe to run even if columns have already been removed.

### FIX-04 — Missing Signup Trigger (CRITICAL — Infinite Onboarding Loop)
**Files:**
- `saas-platform/supabase/migrations/20260302_signup_trigger.sql` (new)
- `saas-platform/src/app/api/onboarding/complete/route.ts` (updated)

**Bug:** The `handle_new_user` database trigger defined in the architecture spec was never implemented in a migration. When a user signed up, **no `user_profiles` row was created**. The onboarding route then ran:
```sql
UPDATE user_profiles SET onboarding_completed_at = ... WHERE id = <user>
```
This matched 0 rows. Supabase/PostgREST returns **no error** for a 0-row UPDATE. The route returned `{success: true}`, the client navigated to `/dashboard`, the dashboard layout queried `user_profiles → null → !null?.onboarding_completed_at = true → redirect('/onboarding')`. **Infinite loop — no user could ever reach the dashboard.**

Three sub-bugs combined to cause this:
1. Missing `handle_new_user` trigger — no `user_profiles` row at signup
2. Route used `UPDATE` instead of `UPSERT` — silent 0-row failure
3. Route swallowed the error with `console.error` and returned `{success:true}` anyway

**Fixes applied:**
1. Created migration `20260302_signup_trigger.sql` with `handle_new_user()` function that creates `user_profiles` + `subscriptions` + `notification_preferences` rows on every new auth signup. Uses `ON CONFLICT DO NOTHING` for safety. Wrapped in `EXCEPTION` block so it can never block signup.
2. Changed profile update in the route from `UPDATE` → `UPSERT` with `onConflict: 'id'` so it creates the row if trigger didn't run (defensive for existing users + local dev)
3. Changed error handling to return HTTP 500 instead of logging and continuing
4. Added subscription `UPSERT` that sets `trial_ends_at = now + 14 days` when onboarding completes — this starts the trial clock properly
5. Added check-first logic for businesses (select before insert) to handle double-submits gracefully

**⚠️ IMPORTANT — Action required:** Run the `20260302_signup_trigger.sql` migration in your Supabase project via the SQL Editor or `supabase db push`. For **existing users** who already signed up but have no `user_profiles` row, run this backfill SQL:
```sql
INSERT INTO public.user_profiles (id, email, created_at, updated_at)
SELECT id, email, created_at, now()
FROM auth.users
WHERE id NOT IN (SELECT id FROM public.user_profiles)
ON CONFLICT (id) DO NOTHING;
```
